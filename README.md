# Next.js + Supabase Auth

A full-stack authentication setup using Next.js 15 App Router, Supabase Auth, and Prisma with SQLite.

---

## Tech Stack

- **Next.js 15** — App Router, Server Components, Server Actions
- **Supabase Auth** — user management, sessions, JWT tokens
- **@supabase/ssr** — SSR-safe Supabase client for Next.js
- **Prisma** — ORM with better-sqlite3 adapter
- **SQLite** — local database for app data
- **TypeScript** — full type safety
- **Tailwind CSS** — styling

---

## Why Supabase Auth over NextAuth

| | NextAuth | Supabase Auth |
|---|---|---|
| User table | You manage it | Supabase manages it |
| Password hashing | You do it with bcrypt | Supabase handles it |
| Session management | You configure JWT | Supabase handles it |
| Code needed | More | Much less |
| Control | More | Less |
| OAuth | Manual setup | Built in |

With NextAuth you own everything. With Supabase Auth you own nothing auth-related — just call their SDK.

## Supabase Client Setup

The most important concept — you need **two separate clients** in Next.js:

### Why two clients?

| | Browser Client | Server Client |
|---|---|---|
| Runs in | Browser only | Server only |
| Session stored in | localStorage | Cookies |
| Used in | Client Components | Server Components, Server Actions, Middleware |
| Import from | `@supabase/ssr` createBrowserClient | `@supabase/ssr` createServerClient |

Next.js runs on both server and browser. The server has no `localStorage` — the only way to persist sessions on the server is cookies. That's why you need two clients.

### `src/lib/supabase/client.ts` — browser client

```ts
import { createBrowserClient } from "@supabase/ssr"

export const createClient = () =>
  createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!
  )
```

Simple — just needs the URL and key. Supabase handles session storage in `localStorage` automatically.

### `src/lib/supabase/server.ts` — server client

```ts
import { createServerClient } from "@supabase/ssr"
import { cookies } from "next/headers"

export const createClient = async () => {
  const cookieStore = await cookies()

  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!,
    {
      cookies: {
        getAll() {
          return cookieStore.getAll()
        },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            )
          } catch {}
        },
      },
    }
  )
}
```

More complex because it manually handles cookies:

- `cookies()` from `next/headers` — reads/writes the request cookies
- `getAll` — reads all cookies so Supabase can find the existing session
- `setAll` — writes refreshed session cookies back to the response
- `try/catch` in `setAll` — Server Components can't set cookies (only Server Actions and Route Handlers can), so it silently fails when called from a Server Component
- `async` — because `cookies()` in Next.js 15 is async

## Prisma Setup

Supabase manages users — Prisma manages your app data (expenses, posts, etc).

### Why use Prisma alongside Supabase?

Supabase has its own database (PostgreSQL) but we use SQLite locally with Prisma for simplicity. In production you could point Prisma at your Supabase PostgreSQL database instead.

### `prisma/schema.prisma`

```prisma
generator client {
  provider = "prisma-client-js"
  output   = "../src/generated/prisma"
}

datasource db {
  provider = "sqlite"
  url      = env("DATABASE_URL")
}

model Expense {
  id        Int      @id @default(autoincrement())
  title     String
  amount    Float
  category  String
  userId    String
  date      DateTime @default(now())
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

Notice `userId` is `String` not `Int` — Supabase user IDs are UUIDs.

No `User` model — Supabase manages users on their side.

### `src/lib/prisma.ts` — singleton

```ts
import { PrismaClient } from "../generated/prisma/client"
import { PrismaBetterSqlite3 } from "@prisma/adapter-better-sqlite3"

const globalForPrisma = globalThis as { prisma?: PrismaClient }

const adapter = new PrismaBetterSqlite3({ url: process.env.DATABASE_URL! })

export const prisma = globalForPrisma.prisma ?? new PrismaClient({ adapter })

if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma
```

Singleton pattern prevents too many database connections during hot reload in development.

### `next.config.ts`

```ts
const nextConfig = {
  serverExternalPackages: ["better-sqlite3"],
}

export default nextConfig
```

`better-sqlite3` is a native Node.js module with C++ bindings. Without this Next.js tries to bundle it for the browser and crashes with `Module not found: Can't resolve 'fs'`.

## Auth Actions

`src/actions/auth.ts` — all auth logic in Server Actions:

```ts
"use server"

import { createClient } from "@/lib/supabase/server"
import { redirect } from "next/navigation"

export const register = async (data: {
  name: string
  email: string
  password: string
}) => {
  const supabase = await createClient()

  const { error } = await supabase.auth.signUp({
    email: data.email,
    password: data.password,
    options: {
      data: {
        name: data.name,
      },
    },
  })

  if (error) return { error: error.message }

  redirect("/dashboard")
}

export const login = async (data: {
  email: string
  password: string
}) => {
  const supabase = await createClient()

  const { error } = await supabase.auth.signInWithPassword({
    email: data.email,
    password: data.password,
  })

  if (error) return { error: error.message }

  redirect("/dashboard")
}

export const logout = async () => {
  const supabase = await createClient()
  await supabase.auth.signOut()
  redirect("/")
}
```

### How each function works:

**`register`**
- `signUp` creates a new user in Supabase Auth — no bcrypt, no manual password hashing
- `options.data` stores extra metadata like name on the user object
- Returns `{ error }` on failure so the form can show the error
- Redirects to dashboard on success

**`login`**
- `signInWithPassword` verifies credentials against Supabase Auth
- Supabase creates a session and stores it in cookies automatically
- Redirects to dashboard on success

**`logout`**
- `signOut` clears the session cookie
- Redirects to home

### Compare to NextAuth:

```
NextAuth register:                    Supabase register:
- hash password with bcrypt           - just call signUp()
- create user in your DB              - Supabase creates the user
- NextAuth creates session            - Supabase creates session
- 20+ lines of code                   - 5 lines of code
```

## Auth Pages

Pages stay as Server Components. Forms are Client Components colocated in the same folder — only move to `components/` if used in multiple places.

### Register

`src/app/(auth)/register/page.tsx`:
```tsx
import RegisterForm from "./register-form"

const RegisterPage = () => {
  return (
    <div>
      <p>This is Register</p>
      <RegisterForm />
    </div>
  )
}

export default RegisterPage
```

`src/app/(auth)/register/register-form.tsx`:
```tsx
"use client"

import { useState, useTransition } from "react"
import { register } from "@/actions/auth"
import Link from "next/link"

const RegisterForm = () => {
  const [name, setName] = useState("")
  const [email, setEmail] = useState("")
  const [password, setPassword] = useState("")
  const [error, setError] = useState("")
  const [isPending, startTransition] = useTransition()

  const handleSubmit = (e: React.FormEvent) => {
    e.preventDefault()
    setError("")

    startTransition(async () => {
      const result = await register({ name, email, password })
      if (result?.error) setError(result.error)
    })
  }

  return (
    <form onSubmit={handleSubmit}>
      {error && <p>{error}</p>}
      <input type="text" placeholder="Name" value={name} onChange={(e) => setName(e.target.value)} />
      <input type="email" placeholder="Email" value={email} onChange={(e) => setEmail(e.target.value)} />
      <input type="password" placeholder="Password" value={password} onChange={(e) => setPassword(e.target.value)} />
      <button type="submit" disabled={isPending}>
        {isPending ? "Registering..." : "Register"}
      </button>
      <Link href="/login">Already have an account? Login</Link>
    </form>
  )
}

export default RegisterForm
```

**Why `useTransition`?**

Gives you `isPending` — true while the Server Action is running. Lets you disable the button and show a loading state without a separate `useState` for loading.

## Accessing the Session

### In a Server Component

```tsx
import { createClient } from "@/lib/supabase/server"

const DashboardPage = async () => {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()

  return (
    <div>
      <p>Welcome {user?.user_metadata?.name}</p>
      <p>Email: {user?.email}</p>
    </div>
  )
}
```

### In a Client Component

```tsx
"use client"

import { createClient } from "@/lib/supabase/client"
import { useEffect, useState } from "react"

const Navbar = () => {
  const [user, setUser] = useState(null)

  useEffect(() => {
    const supabase = createClient()
    supabase.auth.getUser().then(({ data: { user } }) => setUser(user))
  }, [])

  return <p>Welcome {user?.user_metadata?.name}</p>
}
```

### `getUser()` vs `getSession()`

Always use `getUser()` not `getSession()`:

- `getSession()` — just reads the cookie, doesn't verify with Supabase server. Someone could forge a session cookie.
- `getUser()` — always verifies the token with Supabase server. Secure.

## Middleware (Route Protection)

`src/proxy.ts` — runs before every matched request:

```ts
import { createServerClient } from "@supabase/ssr"
import { NextRequest, NextResponse } from "next/server"

export default async function proxy(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request })

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll()
        },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value }) => {
            request.cookies.set(name, value)
          })
          supabaseResponse = NextResponse.next({ request })
          cookiesToSet.forEach(({ name, value, options }) => {
            supabaseResponse.cookies.set(name, value, options)
          })
        },
      },
    }
  )

  const { data: { user } } = await supabase.auth.getUser()

  const isAuthPage =
    request.nextUrl.pathname.startsWith("/login") ||
    request.nextUrl.pathname.startsWith("/register")
  const isDashboard = request.nextUrl.pathname.startsWith("/dashboard")

  if (user && isAuthPage) return NextResponse.redirect(new URL("/dashboard", request.url))
  if (!user && isDashboard) return NextResponse.redirect(new URL("/login", request.url))

  return supabaseResponse
}

export const config = {
  matcher: ["/dashboard/:path*", "/login", "/register"],
}
```

### Why middleware needs its own Supabase client

Middleware doesn't have access to `cookies()` from `next/headers` — it only has the raw `request` object. So we create a Supabase client manually with `request.cookies`.

### The `supabaseResponse` pattern

This is critical. Supabase automatically refreshes expired session tokens on every request. When it refreshes it needs to write new cookies back to the browser.

- `getAll` — reads session cookie from incoming request
- `setAll` — writes refreshed session cookies to both `request` and `supabaseResponse`
- Return `supabaseResponse` — carries the refreshed cookies back to the browser

If you return `NextResponse.next()` instead of `supabaseResponse` the refreshed cookies are lost and users get randomly logged out.

### Redirect logic

```
user + isAuthPage   → redirect to /dashboard  (already logged in, no need for login page)
!user + isDashboard → redirect to /login      (not logged in, can't access dashboard)
everything else     → return supabaseResponse  (continue normally)
```

### The matcher

```ts
matcher: ["/dashboard/:path*", "/login", "/register"]
```

Only run middleware on these routes. Skips static files, images, fonts — much more efficient.

> **Note:** In Next.js 15 the middleware file is named `proxy.ts` and the exported function must be named `proxy` or be a default export.

## Key Differences: Next.js 15 vs older versions

| | Older Next.js | Next.js 15 |
|---|---|---|
| Middleware file | `middleware.ts` | `proxy.ts` |
| `params` | `{ id: string }` | `Promise<{ id: string }>` |
| `searchParams` | `{ q: string }` | `Promise<{ q: string }>` |
| `cookies()` | sync | async |

## Common Errors & Fixes

### `Module not found: Can't resolve 'fs'`
Add to `next.config.ts`:
```ts
const nextConfig = {
  serverExternalPackages: ["better-sqlite3"],
}
```

### `Failed to fetch RSC payload`
Make sure middleware returns `supabaseResponse` not `NextResponse.next()`.

### Users getting randomly logged out
Same issue — return `supabaseResponse` from middleware so refreshed session cookies reach the browser.

### `Proxy is missing expected function export`
In Next.js 15 the function must be named `proxy` or be a default export:
```ts
export default async function proxy(request: NextRequest) { ... }
```

### Params type error
In Next.js 15 params is a Promise:
```ts
// wrong
{ params }: { params: { id: string } }

// correct
{ params }: { params: Promise<{ id: string }> }
// then inside:
const { id } = await params
```
